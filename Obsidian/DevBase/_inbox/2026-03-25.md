### 🔴 1. Crítico: Objetos Recriados e Quebra do `useMemo` no Hook de Modais
**Arquivo:** [useManualPaymentModals.ts](cci:7://file:///c:/Projetos/multipay-admin-site-web/src/presentation/hooks/useManualPaymentModals.ts:0:0-0:0) e [ManualPayment.tsx](cci:7://file:///c:/Projetos/multipay-admin-site-web/src/presentation/components/forRoutes/order/details/components/manualPayment/ManualPayment.tsx:0:0-0:0)

**O Problema:**
O hook [useManualPaymentModals](cci:1://file:///c:/Projetos/multipay-admin-site-web/src/presentation/hooks/useManualPaymentModals.ts:19:0-88:1) retorna um novo objeto `{ modals: {...}, selectedData: {...} }` em absolutamente **toda renderização**. Na interface [ManualPayment.tsx](cci:7://file:///c:/Projetos/multipay-admin-site-web/src/presentation/components/forRoutes/order/details/components/manualPayment/ManualPayment.tsx:0:0-0:0), a variável `mappedAccordionData` é memoizada, porém utiliza a dependência `modals` (`[..., modals]`).
Como esse `modals` nunca mantém idêntica referência, o **`useMemo` é inofensivo/inútil** e quebra, fazendo as tabelas de dados recalcularem na íntegra no menor evento (ex: ao abrir ou fechar os modais).

**Como Corrigir:**
No hook [useManualPaymentModals](cci:1://file:///c:/Projetos/multipay-admin-site-web/src/presentation/hooks/useManualPaymentModals.ts:19:0-88:1), utilize `useCallback` nas funções e `useMemo` no retorno:
```tsx
const openApprove = useCallback((payment: IManualPayment) => {
  setSelectedPayment(payment);
  setApproveModalOpen(true);
}, []);

// Faça o mesmo para outras funções...

return useMemo(() => ({
  modals: { /* ... */ },
  selectedData: { /* ... */ }
}), [approveModalOpen, rejectModalOpen, /* demais estados primitivos */, selectedPayment]);
```

### 🟠 4. Importante: Ausência de `React.memo` e Funções Inline nas Rows
**Arquivos:** [ManualPayment.tsx](cci:7://file:///c:/Projetos/multipay-admin-site-web/src/presentation/components/forRoutes/order/details/components/manualPayment/ManualPayment.tsx:0:0-0:0) e [ManualPaymentRow/index.tsx](cci:7://file:///c:/Projetos/multipay-admin-site-web/src/presentation/components/forRoutes/order/details/components/manualPayment/ManualPaymentRow/index.tsx:0:0-0:0)

**Como Corrigir:**
1. Envolva em um memo: `export const ManualPaymentRow = React.memo(({ ... }: Props) => { ... })`
2. ## SKILL DE PERFORMANCE

Com base na análise realizada, identifiquei que o projeto possui sérios gargalos no tamanho do bundle e na eficiência do carregamento. O uso  
  de duas bibliotecas de UI (MUI e Ant Design) em conjunto com o Moment.js e o abuso de barrel imports do MUI são as causas principais.

  Abaixo está o relatório detalhado conforme solicitado.

  📊 Análise de Performance — LAB-multipay-admin-site-web
  Stack: Next.js 15.3.8 + MUI 6.4.5 + Ant Design 5.24.5

  Resumo Executivo
  O projeto apresenta um bundle front-end excessivamente pesado devido à coexistência de dois frameworks de UI (MUI e AntD) e à falta de        
  otimização nos imports de componentes e ícones. A dependência do Moment.js agrava o peso do bundle final. Quase todas as páginas principais   
  são marcadas como 'use client', enviando todo esse JavaScript para o navegador do usuário.

  🔴 Problemas Críticos
   - MUI Barrel Imports: Foram encontradas 69 ocorrências de imports destruturados diretamente de @mui/material. Sem modularizeImports no       
     next.config, isso força o Next.js a incluir quase todo o pacote MUI no bundle de cada página.
     - Localização: Diversos arquivos (ex: src/presentation/components/forRoutes/home/components/filter/Formulary.tsx)
     - Impacto: Adiciona ~400-600 KB (sem gzip) ao bundle principal desnecessariamente.
   - Frameworks de UI Duplicados: O projeto utiliza tanto o MUI (v6) quanto o Ant Design (v5). O Ant Design é usado apenas para componentes     
     pontuais como Typography e Tooltip.
     - Localização: src/presentation/layouts/default/globalProviders/GlobalProviders.tsx e componentes de layout.
     - Impacto: Carregamento de dois runtimes de estilo e bibliotecas de componentes massivas simultaneamente.

  🟡 Melhorias Recomendadas
   - Dependência Moment.js: O moment é conhecido por ser pesado e não amigável a tree-shaking. Recomenda-se substituir por date-fns ou dayjs.
     - Localização: src/presentation/components/forRoutes/order/refund/components/PaymentComponent/index.tsx → Impacto: Redução de ~67 KB
       (gzipped).
   - Uso de Tags <img> Nativas: Diversos componentes usam <img> em vez de next/image, perdendo otimização automática de formato (WebP) e lazy
     loading.
     - Localização: src/presentation/components/forRoutes/order/details/components/components/AttachmentsModal/index.tsx
   - Múltiplas Bibliotecas de Ícones: O projeto utiliza @iconify/react, @mui/icons-material e react-icons. Consolidar em apenas uma reduziria o
     tamanho do bundle e a complexidade.
   - Falta de modularizeImports: O arquivo next.config.ts não possui a configuração de transformação de imports para MUI e Ícones.

  🟢 Boas Práticas Encontradas
   - Next.js Fontes: Uso correto do next/font/google para a fonte Inter no layout principal.
   - Arquitetura de Pasta: Separação clara entre domain, infra e presentation.
   - Next.js App Router: Estrutura moderna que permite a transição para Server Components.

- ferramentas de análise de bundle.

Em termos simples: **Tree-shaking é o processo de remover código morto (não utilizado) do seu bundle final.**

Imagine uma árvore real. A árvore é o seu código completo (incluindo todas as bibliotecas que você instalou). O Tree-shaking é o ato de **sacudir** essa árvore para que as folhas secas (as funções e componentes que você importou mas nunca chamou no código) caiam, deixando apenas o que é vital para a aplicação rodar.
